バイク部！ アプリ

プロジェクト概要

開発人数: 1人

担当フェーズ: 企画・設計・実装・テスト

開発期間:約7ヶ月(働きながら地道に)

使用技術: Flutter,Firebase,GoogleMapAPI,admob

どんなアプリ？: バイク乗りのためのSNSアプリです。ツーリングの募集やツーリングスポット情報の検索、投稿ができます。mapからアプリを利用している近くのユーザーと交流することもできます。

なぜこのプロジェクトを作ったのか

本アプリ、バイク部！は、以下の3つの主要な目的を持って開発しました：

1.個人の趣味を活かしたサービス開発

バイクという自分の趣味を活かしてサービスを作ることで、ユーザーのニーズをより深く理解し、本当に役立つ機能を提供できると考えました。自分自身が欲しいと思う機能を実装することで、他のバイク乗りにも喜んでもらえるアプリを作りたいという思いがこのプロジェクトの原動力となりました。

2.技術力の向上

仕事でFlutterを初めて触り、もっと深く学びたい、自分でもプロダクトを作成したいと思いこのアプリを作成しました。このプロジェクトを通じて、Flutterの高度な機能や最新のベストプラクティスを学び、スキルを磨きたいと考えました。

3.ライダー同士のつながりを増やしたい

ツーリングは一人でも楽しいものですが、趣味の合う仲間と一緒に走るともっと楽しくなります。私自身、長らく一人でツーリングに行くことが多く、その楽しさを知る一方で、仲間と一緒に走る経験もあり、その素晴らしさを実感していました。しかし、気の合う仲間を見つけることの難しさも感じていました。このアプリを通じて、同じ趣味を持つライダー同士が簡単につながり、新しい仲間を見つけられる場を提供したいと考えました。一人の時間も大切にしながら、時には気の合う仲間と走る喜びを、より多くのライダーに体験してもらいたいという思いがこのアプリの原点です。

プロジェクト構成

バイク部！アプリは、オニオンアーキテクチャを採用しています。

  1. プレゼンテーション層: UI/UX、ユーザーとのインタラクション
  2. アプリケーション層: ユースケース、アプリの設定など
  3. ドメイン層: エンティティ、ビジネスロジック
  4. インフラストラクチャ層: データベース、外部サービス

アーキテクチャ比較

1. オニオンアーキテクチャ（選択したアーキテクチャ）

メリット：

  • 依存関係が内側に向かうため、外部の変更が内部に影響しにくい(データベースなどの変更がビジネスロジックに影響与えにくい)
  • ドメインロジックが中心にあり、ビジネスロジックの変更が容易
  • テスト容易性が高い
  • 将来的な技術スタックの変更に強い

デメリット：

  • 小規模プロジェクトでは過剰に複雑になる可能性がある
  • 学習コストが高い

2. MVC (Model-View-Controller)

メリット：

  • シンプルで理解しやすい
  • 多くの開発者が慣れ親しんでいる
  • 小規模から中規模のプロジェクトに適している

デメリット：

  • コントローラーが肥大化しやすい
  • ビジネスロジックとプレゼンテーションロジックの分離が不十分になりがち
  • テスト容易性が低い場合がある

3. MVVM (Model-View-ViewModel)

メリット：

  • UIとビジネスロジックの分離が明確
  • データバインディングによる効率的なUI更新
  • テスト容易性が高い

デメリット：

  • MVCよりも複雑
  • 小規模プロジェクトでは過剰になる可能性がある

4. クリーンアーキテクチャ

メリット：

  • 高度に分離された構造で、各層の独立性が高い
  • テスト容易性が非常に高い
  • フレームワークに依存しない設計が可能

デメリット：

  • 構造が複雑で、小規模プロジェクトには過剰
  • 開発速度が遅くなる可能性がある
  • 学習コストが高い

オニオンアーキテクチャ選択理由

  1. 将来的な拡張性： バイク部！アプリは、将来的な機能追加や技術スタックの変更を見据えています。オニオンアーキテクチャは、外部層の変更が内部に影響しにくい構造を持つため、この要件に適しています。
  2. データベース置換の可能性： ユーザー増加に伴い、将来的なデータベースの置換を想定しています。オニオンアーキテクチャでは、インフラストラクチャ層を変更しても、ドメイン層やアプリケーション層に影響を与えにくいため、この要件に合致します。
  3. ビジネスロジックの中心性： バイクライダー向けSNSとして、複雑なビジネスロジック（ツーリング管理、位置情報処理など）が存在します。オニオンアーキテクチャは、ドメインロジックを中心に置くため、これらの処理を効果的に管理できると考えました。
  4. テスト容易性： オニオンアーキテクチャは、各層が明確に分離されているため、単体テストや統合テストが容易です。
  5. 適度な複雑性： MVCよりは複雑ですが、クリーンアーキテクチャほどの複雑さはありません。バイク部！アプリの規模と将来の拡張性を考慮すると、オニオンアーキテクチャはの複雑さは適切と考えました。

結論

バイク部！アプリの要件、将来の拡張性、データベース置換、保守性、テスト容易性を考慮した結果、オニオンアーキテクチャが最適な選択肢であると判断しました。この選択により、アプリケーションの長期的な成長と品質維持が可能になると考えています。

技術的な挑戦と解決策

1. データベース設計と最適化

課題: NoSQLデータベース（Firebase Firestore）を使用しながら、効率的なデータ構造の設計

解決策: データの正規化と非正規化のバランスを取り、頻繁にアクセスされるデータの読み取り効率を向上

2. パフォーマンスとコスト最適化

課題: アプリの応答性を維持しつつ、Firebase利用コストを抑制

解決策:

  • ページング処理の実装
  • キャッシュの効果的な利用
  • クエリの最適化

    • 3. リアルタイム位置情報処理

    課題: Google Maps APIを使用した効率的な位置情報の更新と表示

    解決策:位置情報更新の最適化、ジオハッシュを用いた効率的な近隣ユーザー検索

    4. クロスプラットフォーム開発

    課題: iOS/Android両プラットフォームでの一貫した体験提供

    解決策: 各OSで実機を用いたE2Eテスト、プラットフォーム固有の機能に対する適切な抽象化

    5. スケーラビリティ

    課題: ユーザー数と投稿数の増加に伴う系統的なパフォーマンス低下の防止

    解決策: クラウドファンクションの活用、効率的なインデックス設計、必要に応じたサーバーレス機能の導入

    今後の展望

    • 投稿画面でのソート・検索機能: 現在は投稿数も少なく問題ないが、投稿数増加に伴いソート・検索機能の実装は必須。場合によってはデータベースの置換も検討。
    • チャット機能の向上:現在は費用削減のため簡易チャットの形式で行っているが収益増加具合によりチャット機能を改善する
    • ツーリングスポットの投稿情報の一覧化:現在各スポットに結び付く形でスポットの投稿情報を表示しているが一覧で表示する
